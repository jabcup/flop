Contexto

El contexto

Un videojuego de póquer multijugador Texas Hold'em, con cliente de escritorio y backend con API REST, construido para la materia de Tecnología de Bases de Datos.

→ Cliente PyGame de escritorio, multijugador por turnos asíncronos.
→ Backend FastAPI que expone endpoints REST sobre el estado de cada mano.
→ SQL Server como fuente de verdad persistente para usuarios, mesas y manos jugadas.
→ Procedimientos almacenados y triggers centralizan la lógica de datos.

Persistencia
Durante la mano
Mazo, turno y pot viven en memoria en el backend. No se consulta la BD por cada acción.
Al cerrar la mano
Se persiste todo de golpe: jugadores, cartas, acciones y resultados, con un único UPDATE.
Consultas del cliente
El cliente hace polling cada X segundos para conocer el estado actual de la mesa.

07 / 10
Decisiones

Decisiones de diseño

Decisión 01
El ganador no se guarda explícitamente.
Se deduce comparando cartas con treys al cerrar la mano, leyendo usuarios_mano.
Decisión 02
Cartas comunitarias en formato treys.
Ejemplo: Qs, 4h. Campos nvarchar(3).
Decisión 03
El estado temporal vive solo en memoria.
Mazo, turno y pot nunca tocan la BD hasta que la mano termina.
Decisión 04
Número de tarjeta guardado en recargas.
Para manejo de disputas y trazabilidad de pagos.
Decisión 05
Tutoriales hardcodeados en el cliente.
Viven en PyGame para reducir complejidad de la BD y del backend.
Decisión 06
Persistencia en bloque al final de la mano.
Un único UPDATE cierra la mano y vuelca todo el estado acumulado.
